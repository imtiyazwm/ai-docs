# WaveMaker AI Release Notes

Use this skill to help a writer or developer fill in a versioned release notes file for WaveMaker AI. The base file is generated by another tool — your job is to guide the user through adding entries correctly, one item at a time, and to catch anything that violates our style or structure before it ships.

## When to use

- User wants to add a feature, enhancement, or bug fix to a release notes file.
- User asks how to write a release note entry or what category something belongs in.
- User wants to verify or correct doc links in a release notes file.
- User is reviewing a draft release notes file for errors.

## When NOT to use

- User wants a public-facing post celebrating a feature → use the `wm-ai-feature-announcements` skill.
- User wants a narrative blog post or engineering story → use the blog skill.
- User wants a reference or how-to doc page → use the `wm-ai-create-guide` skill.

## Inputs to collect

Before editing the file, confirm:

1. **Which version** — default to the latest version found in `docs/release-notes/`. Resolve the file path automatically (see Step 0). Confirm with the user before proceeding.
2. **What was shipped** — feature, enhancement, or bug fix? Let the user describe it in their own words; you will help categorise it.
3. **Which tab** — Features, Enhancements, or Bug Fixes. See conventions for definitions.
4. **Which accordian** — User Interface, Backend, Platform, or Product Ecosystem. See conventions for definitions.
5. **Platform pills** — does this apply to Web, Mobile, or both? Or neither (no pill)?
6. **Doc link** — is there an existing doc to link to? If yes, get the path and verify it exists. If not, ask whether one needs to be created first.

Do not add a placeholder link. If a doc does not exist yet, tell the user to create it first.

## File layout

Release notes files follow this path pattern:

```text
docs/release-notes/release-version-<N>/<version-dir>/<version>.mdx
```

For example:

```text
docs/release-notes/release-version-1/version-1-0-x/1.0.0.mdx
```

The top-level template used to generate a new file is `assets/release-notes-template.mdx`.

## Procedure

### Step 0 — Resolve the target file

Before asking the user anything, find the latest release notes file:

1. List `docs/release-notes/` to find the highest `release-version-<N>` directory.
2. Within that directory, find the highest version subdirectory and then the `.mdx` file inside it.
3. Tell the user which file you found (e.g., "I'll be editing `docs/release-notes/release-version-1/version-1-0-x/1.0.0.mdx` — is that the right version?").
4. If the user wants a different version, ask them to name it and resolve accordingly.

Do not proceed until the user has confirmed the target file.

### Step 1 — Identify the item

Ask the user: "What did you ship? Tell me the feature, enhancement, or fix — I'll help place and write it."

From their description, determine:

- **Tab**: Feature = new capability, Enhancement = improvement to something existing, Bug Fix = something broken that now works.
- **Accordian**: UI work → User Interface; APIs, data sources, Java backend → Backend; AI agents, platform-level tooling → Platform; docs, Academy, Storybook, Marketplace → Product Ecosystem.

If the classification is ambiguous, make a recommendation and explain why. Ask for confirmation before proceeding.

### Step 2 — Draft the entry

Write a draft entry using the format in `references/conventions.md`. Keep the description to **one sentence**. If the user's input is multi-sentence, trim it and suggest what the extra detail can link to.

### Step 3 — Handle the doc link

Ask: "Is there an existing doc to link to, or does this need one?"

- **Existing doc**: ask for the path, then verify the file exists in the repo using `find` or `ls`. Resolve the path relative to the release notes file. If the file is missing, warn the user and do not add the link.
- **New doc needed**: tell the user the doc must be created before the release note references it. Do not add a placeholder link.
- **External URL**: accept as-is (Academy, Storybook, Marketplace, external blog).
- **No link needed**: fine — not every entry requires one. Short, self-explanatory bug fixes can stand alone.

### Step 4 — Confirm before writing

Before touching any file, present the complete planned entry to the user:

- The target file path
- Which tab and accordian it will go into
- The full MDX entry (title, pills, body, link if any)

Ask: "Does this look right? I'll add it once you confirm."

Only edit the file after explicit confirmation. If the user wants changes, update the draft and confirm again.

### Step 5 — Add to the file

Edit the release notes file and insert the entry in the correct accordian, maintaining the list structure. See `references/conventions.md` for the exact format.

### Step 6 — Repeat or finish

Ask if there are more items. When the user is done, run a final scan of the file and flag:

- Body copy longer than one sentence.
- Relative links that include a `.md` or `.mdx` extension.
- Relative links where the target file does not exist.
- Any tab or accordian name that is not in the fixed set.

## Common mistakes to avoid

- **Multi-sentence body** — the body is one sentence. Everything else goes in a linked doc.
- **Placeholder links** — do not write `[Documentation](#)` or `[link to be added]`. Verify first, add only when confirmed.
- **Link with file extension** — relative links must not include `.md` or `.mdx`. Strip the extension before writing.
- **Broken relative link** — resolve the path from the `.mdx` file and confirm the target exists. If it doesn't, stop.
- **Adding a new tab or accordian** — the structure is fixed. Never introduce a new tab or section name.
- **Putting UI copy in the wrong accordian** — a frontend widget fix belongs in User Interface, not Platform, even if it involved backend changes.
- **Marketing tone** — describe what the capability does, not how impressive it is.
- **Missing pill** — if a feature is web-only or mobile-only, include the appropriate pill. Omit only when platform-neutral.

## Validation checklist

- [ ] Entry is in the correct tab (Features / Enhancements / Bug Fixes).
- [ ] Entry is in the correct accordian (User Interface / Backend / Platform / Product Ecosystem).
- [ ] Title is a `###` heading, 4–7 words.
- [ ] Body is exactly one sentence.
- [ ] Platform pills are correct and present where needed.
- [ ] Any relative link has been verified to exist in the repo.
- [ ] Relative links have no `.md` or `.mdx` extension.
- [ ] No placeholder or unverified links remain.
- [ ] No new tabs or accordian titles were introduced.
- [ ] Empty tabs use the self-closing form `<TabItem name="..." />`.
